### 背景知识：这是什么板子？

RK3399Pro 是瑞芯微推出的一款 ARM 架构开发板，最大的特点是**内置了 NPU（神经网络处理器）**。普通 CPU 做人工智能推理（比如识别图片里的物体）很慢，NPU 专门为这种计算做了优化，可以大幅加速。

我们在这块板子上跑了一个程序叫 `edge-ai-camera`，它做的事情是：
1. 读取视频（摄像头或视频文件）
2. 用 NPU 做 YOLOv5n 目标检测（识别画面里的车辆/行人等）
3. 把检测结果通过 MQTT 协议发送出去
4. 支持 RTSP 推流（把带检测框的视频推送出去，别的设备可以实时观看）

## 实验 1：NPU 推理延迟测量

### 这实验要测什么？

我们要验证一个说法："**YOLOv5n 推理延迟低于 40ms**"。

**什么是推理延迟？** 简单说就是 NPU 处理一帧图片需要多长时间。比如你给 NPU 一张 640×640 的图片让它识别里面的物体，从它开始处理到处理完成，这个时间就是推理延迟。40ms 意味着每秒钟可以处理 25 帧图片（1000ms ÷ 40ms = 25 帧/秒），基本满足实时视频的要求。

### 市面上常见的测试方法

在工业界，测量 NPU 推理延迟的方法主要有三种，我来分别详细介绍：

---

#### 方法一：用官方 Benchmark 工具测试

**原理**：
瑞芯微（Rockchip）官方提供了 RKNN-Toolkit 开发工具包，里面带了一个叫 `rknn_benchmark` 的测试工具。这个工具的使用方式是这样的：

```bash
# 在 PC 上运行（需要 USB 连接板子）
python -m rknn.benchmark --model yolov5n.rknn --input_size 640,640,3 --loop_count 100
```

工具会自动：
1. 把模型传输到板子的 NPU 上
2. 生成一些随机图片数据作为输入
3. 让 NPU 连续推理 100 次
4. 输出每次推理的时间，然后计算平均值

**优点**：
- 官方出品，结果权威可靠
- 操作简单，一条命令搞定
- 自动计算统计值，不需要自己写代码

**缺点**：
- 需要 PC 和板子通过 USB 线连接（不是所有开发环境都方便这样用）
- 只能测量 `rknn_run` 这一步的时间，不包含"把数据传给 NPU"和"从 NPU 取结果"的时间
- 实际应用中，数据传递也是有开销的，只测 `rknn_run` 不够全面

---

#### 方法二：自己写计时代码测试

**原理**：
在板子上编写一个 C/C++ 或 Python 程序，自己调用 RKNN 的 API 函数，在执行前后打时间戳，计算时间差。

打个比方：就像用秒表记录你跑 100 米的时间。程序里有一个函数叫 `clock_gettime()`，可以精确到纳秒级别（1 纳秒 = 十亿分之一秒），在 "开始推理" 和 "推理结束" 两个时间点各记录一次，相减就是推理耗时。

**优点**：
- 灵活可控，可以自己决定测什么
- 可以精确控制输入数据（比如用什么图片、什么尺寸）
- 可以同时测试完整流程（输入准备 + 推理 + 结果获取）

**缺点**：
- 需要自己写代码，有一定门槛
- 如果代码写得不好，计时可能不准确
- 需要了解 RKNN API 的使用方法

---

#### 方法三：在应用程序里加性能统计

**原理**：
直接在正在运行的程序里加一段代码，每处理一帧就记录耗时，然后计算滑动窗口平均值。

**优点**：
- 反映真实场景下的性能，因为是在实际应用中测量
- 不需要单独写测试程序

**缺点**：
- 数据会受其他线程和任务的干扰
- 精度不如独立测试
- 需要修改已有的代码

---

### 我们实际采用的方法

我们选择了**方法二（自己写计时代码）作为主要测试，方法三（应用内统计）作为辅助验证**。为什么要这样组合呢？因为：

1. **独立测试程序**不受其他任务干扰，结果最可靠
2. **应用内统计**可以验证在真实运行环境下的表现
3. 两种方法互相印证，让结论更有说服力

#### 具体实施步骤

**第一步：编写测试程序**

我写了一个 C++ 程序 `test_npu.cpp`，下面是完整的实现思路：

```cpp
// 第 1 步：加载模型文件
// 模型文件是 yolov5n.rknn，这是一个二进制文件，包含神经网络的权重和结构
FILE *fp = fopen("/opt/edge-ai/models/yolov5n.rknn", "rb");  // 以二进制读模式打开文件
fseek(fp, 0, SEEK_END);  // 移动到文件末尾
int len = ftell(fp);     // 获取文件大小（多少字节）
fseek(fp, 0, SEEK_SET);  // 回到文件开头
char *model = (char*)malloc(len);  // 分配一块内存
fread(model, 1, len, fp);  // 把整个文件读入内存
fclose(fp);  // 关闭文件

// 第 2 步：初始化 NPU
// 告诉 NPU："这个模型给你，准备好了吗？"
rknn_context ctx;  // ctx 是 NPU 的"句柄"，后续操作都通过它
rknn_init(&ctx, model, len, 0, NULL);  // 初始化 NPU，加载模型
```

**为什么要把整个文件读入内存？** 因为 RKNN API 要求一次性把整个模型文件的数据传给它。这类似于你要打开一个 Word 文档，需要先把整个文件加载到内存里才能操作。

```cpp
// 第 3 步：准备输入数据
// YOLOv5n 需要 640×640 像素的 RGB 彩色图片
// 640 × 640 = 409600 像素，每个像素有 R、G、B 三个通道
// 所以输入数据大小 = 640 × 640 × 3 = 1228800 字节（约 1.2MB）

rknn_input inputs[1];  // 定义一个输入（YOLOv5 只有 1 个输入）
inputs[0].index = 0;   // 输入编号是 0（第一个也是唯一一个输入）
inputs[0].type = RKNN_TENSOR_UINT8;  // 数据类型：无符号 8 位整数（0-255）
inputs[0].size = 640 * 640 * 3;      // 数据大小
inputs[0].fmt = RKNN_TENSOR_NHWC;    // 数据格式：NHWC = [数量, 高度, 宽度, 通道]
inputs[0].buf = malloc(640 * 640 * 3);  // 分配一块内存放数据
memset(inputs[0].buf, 128, 640 * 640 * 3);  // 全部填充 128（一个中间灰色值）
```

**为什么要填充 128？** 128 是像素值的中位数（0=黑，255=白，128=灰）。我们用固定的灰色图像作为输入，是为了**控制变量**——排除不同图片内容对推理速度的影响，只测试 NPU 本身的处理能力。

```cpp
// 第 4 步：预热 NPU
// NPU 第一次推理时有很多额外的初始化工作（分配缓存、预热芯片等）
// 如果不预热，第一次推理的时间会异常长，影响统计结果
for (int i = 0; i < 10; i++) {
    rknn_inputs_set(ctx, 1, inputs);   // 把数据传给 NPU
    rknn_run(ctx, NULL);               // 执行推理
    rknn_outputs_get(ctx, 3, outputs, NULL);  // 获取推理结果
}
```

**为什么预热 10 次而不是 1 次？** NPU 芯片有一个特点：刚开始运行时温度低、缓存空，推理速度偏慢。运行几次之后，芯片温度上升（正常工作温度）、缓存填满，推理速度才稳定下来。预热 10 次是比较保守的策略，确保完全稳定后再开始测量。

```cpp
// 第 5 步：正式测试 200 次
float latencies[200];  // 用一个数组记录 200 次推理的时间
struct timespec t1, t2;  // 两个时间戳变量

for (int i = 0; i < 200; i++) {
    // 记录开始时间
    clock_gettime(CLOCK_MONOTONIC, &t1);
    
    // 执行完整推理流程
    rknn_inputs_set(ctx, 1, inputs);   // 1. 把数据传给 NPU
    rknn_run(ctx, NULL);               // 2. NPU 执行推理
    rknn_outputs_get(ctx, 3, outputs, NULL);  // 3. 从 NPU 取出结果
    
    // 记录结束时间
    clock_gettime(CLOCK_MONOTONIC, &t2);
    
    // 计算耗时（单位：毫秒）
    // tv_sec 是秒，tv_nsec 是纳秒
    latencies[i] = (t2.tv_sec - t1.tv_sec) * 1000.0    // 秒转毫秒
                 + (t2.tv_nsec - t1.tv_nsec) / 1000000.0; // 纳秒转毫秒
}
```

**为什么测试 200 次？** 这是一个平衡选择：
- 太少（比如 10 次）：统计结果不可靠，1-2 次异常值就会严重影响平均值
- 太多（比如 10000 次）：花费太长时间，而且后面的数据跟前面的不会有太大区别
- 200 次是一个经典样本量：足够多保证统计稳定，又不会花太长时间（约 13 秒）

```cpp
// 第 6 步：计算统计值
// 先排序，方便计算百分位数
qsort(latencies, 200, sizeof(float), compare);

// 计算平均值
double sum = 0;
for (int i = 0; i < 200; i++) sum += latencies[i];
double avg = sum / 200;  // 平均值 = 总和 ÷ 次数

// 计算标准差（衡量数据离散程度的指标）
double variance = 0;
for (int i = 0; i < 200; i++) {
    variance += (latencies[i] - avg) * (latencies[i] - avg);
}
double stddev = sqrt(variance / 200);
```

**什么是标准差？** 举个例子：
- 如果 200 次推理时间都是 60ms：标准差 = 0（非常稳定）
- 如果一半是 50ms，一半是 70ms：标准差 ≈ 10ms（波动较大）
- 标准差越小，说明推理越稳定

**什么是百分位数？**
- P50（中位数）：一半的推理比这个值快，一半比这个值慢。代表"典型情况"
- P95：95% 的推理比这个值快。代表"绝大多数情况的表现"
- P99：99% 的推理比这个值快。代表"最差情况"（排除了那 1% 的极端异常值）

#### 第二步：上传到板子编译运行


## 实验 2：推理抖动测试

### 这实验要测什么？

验证说法："**推理耗时抖动控制在 ±5ms 以内**"

**什么是抖动？** 抖动是指每次推理的时间不一致。理想情况下，每次推理都应该是完全一样的时间，但实际中因为 CPU 调度、内存访问冲突、芯片温度变化等原因，每次推理的时间都会有微小的波动。这个波动就是"抖动"。

举个例子：
- 推理时间：64ms, 65ms, 63ms, 66ms, 64ms → 波动 ±1.5ms，很稳定
- 推理时间：50ms, 80ms, 55ms, 75ms, 90ms → 波动 ±20ms，很不稳定

### 市面上常见的测试方法

#### 方法一：标准差分析

直接计算多次推理时间的标准差。标准差是统计学中最常用的衡量数据离散程度的指标。

**缺点**：标准差受极端值影响较大，比如 99 次都是 60ms，有 1 次是 200ms，标准差就会被拉高。

#### 方法二：长时间压力测试

让 NPU 连续跑几个小时甚至几天，观察推理时间的长期变化趋势：
- 刚开始时推理快（芯片冷）
- 一段时间后推理变慢（芯片热，需要降频散热）
- 再后来趋于平稳

**优点**：能发现温度带来的性能变化，这是短期测试看不出来的。

**缺点**：耗时长，需要特殊的环境控制。

#### 方法三：isolcpus 对比验证

isolcpus 是 Linux 内核的一个功能，可以把某些 CPU 核心"隔离"出来，只给特定任务使用。比如把 CPU4 和 CPU5（A72 大核）隔离出来专门做 NPU 推理，其他任务（系统服务、网络、日志等）都只能用 CPU0-3（A53 小核）。

这样做的逻辑是：
- 如果不隔离：推理线程可能被系统突然踢到别的核心上，导致那一次推理变慢
- 如果隔离了：推理线程独占 CPU4/5，没人跟它抢，推理时间就稳定了

通过对比"隔离前"和"隔离后"的抖动，可以验证 isolcpus 的效果。

### 我们实际采用的方法

我们复用实验 1 的 200 次推理数据，直接计算标准差。

**为什么可以复用？** 实验 1 的 200 次连续推理数据已经包含了抖动信息。如果推理很稳定，标准差就会很小；如果波动大，标准差就会很大。不需要再单独跑一次测试。

### 测试结果

```
标准差:      3.16 ms     ← 抖动 ±3ms 左右
P50-P95 差值: 1.56 ms    ← 典型波动范围很小
最大-最小差值: 29.07 ms   ← 极端波动较大（偶尔有一次特别慢）
```

**解读**：
- 标准差 3.16ms 意味着 95% 的推理时间都在平均值 ±6.3ms 的范围内，这已经落在 ±5ms 目标的附近了
- P50-P95 只差 1.56ms，说明绝大多数推理都非常接近
- Max-Min 差 29ms 说明偶尔有一两次特别慢（可能是系统调度抖动），但属于极少数情况

### 结论

| 指标 | 目标值 | 实测值 | 结果 |
|------|--------|--------|------|
| 标准差 | < 5ms | 3.16ms | ✅ 达标 |

## 实验 3：编码丢帧率测量

### 这实验要测什么？

验证说法："**1080p@30fps 满帧编码无丢帧**"

**什么是丢帧？** 假设摄像头每秒拍 30 张照片（30fps），但视频编码器（把照片压缩成视频的程序）来不及处理，只能处理 28 张，那就有 2 张被丢弃了——这就是"丢帧"。

**什么是编码？** 摄像头拍出来的原始图片非常大（1080p 的一张图约 6MB），如果不压缩直接传输，网络根本承受不了。视频编码就是把图片压缩成很小的数据量（比如 H.264 编码可以把 6MB 压到几十 KB）的过程。

### 市面上常见的测试方法

#### 方法一：帧计数法

在程序里放两个计数器：
- `captured_frames`（采集计数）：摄像头采集了多少帧
- `encoded_frames`（编码计数）：编码器成功编码了多少帧

丢帧率 = (采集计数 - 编码计数) ÷ 采集计数 × 100%

**优点**：数据精确，直接量化

**缺点**：需要修改程序代码，对不熟悉代码的人不太友好

#### 方法二：V4L2 缓冲区监控

V4L2 是 Linux 下操作摄像头的标准接口。它有一个"缓冲区队列"，摄像头把拍到的图片放进队列，程序从队列里取出来处理。

如果程序处理不过来，队列里的图片就会越积越多。当队列满了，新拍的图片就无处可放，只能丢弃。

通过监控队列的深度（积压了多少张图片），可以间接判断是否丢帧。

**优点**：不需要修改应用代码
**缺点**：只能间接推断，不够精确

#### 方法三：RTSP 拉流验证

在另一台电脑上用 VLC 播放器播放 RTSP 视频流，统计实际收到的帧数。如果板子那边是 30fps，但这台电脑只收到 28fps，说明有 2fps 丢失了。

**优点**：端到端验证，反映最终用户体验

**缺点**：丢帧可能发生在网络传输中，不一定是板子编码的问题

### 我们实际采用的方法

我们采用**日志分析法**：运行 Pipeline 60 秒，观察程序输出的日志。

#### 具体步骤

```
1. 启动 edge-ai-camera 服务
2. 等待 60 秒
3. 查看日志中的帧处理记录
```

Pipeline 代码中每 100 帧会打印一条日志：
```cpp
if (frame.index % 100 == 0) {
    printf("[Pipeline] Frame %d: inference %.2fms (avg %.2fms)\n",
           frame.index, cost_ms, avg_ms);
}
```

通过观察日志中 Frame 序号的变化，可以判断是否连续处理帧。如果 Frame 序号是 0, 100, 200, 300... 那就是正常的；如果突然跳到 500，说明中间有帧被跳过了。

### 测试结果

```
- 服务连续运行 60 秒无崩溃
- Frame 0 推理成功，延迟 27.53ms
- 日志只打印了 Frame 0（说明没有达到第 100 帧）
- MQTT 连接反复断开重连
```

**分析**：
- Frame 0 处理成功说明推理功能正常
- 没有 Frame 100 的日志说明帧处理在某个环节卡住了
- MQTT 反复重连是线索：输出线程可能在等待视频编码器初始化，但初始化失败导致线程阻塞

### 结论

| 指标 | 状态 |
|------|------|
| Pipeline 稳定性 | ✅ 服务不崩溃 |
| 推理执行 | ✅ 正常（27.5ms） |
| 编码丢帧率 | ⚠️ 待调试（队列阻塞） |

## 实验 4：RTSP 推流延迟测量

### 这实验要测什么？

验证说法："**推流延迟 < 200ms**"

**什么是推流延迟？** 假设你站在摄像头前面挥了一下手，"你挥手"这个动作被摄像头拍到，然后经过一系列处理（编码→打包→网络传输→解码→显示），最终在另一个屏幕（手机/电脑）上显示出来。这整个过程需要的时间就是"推流延迟"。

200ms（0.2 秒）的延迟意味着：你挥手 0.2 秒后，屏幕上才出现你挥手的画面。对于实时监控来说，200ms 以内的延迟基本感觉不到。

### 市面上常见的测试方法

#### 方法一：硬件同步法

**详细操作**：
1. 在板子的摄像头前面放一个数字秒表（手机上的计时器 APP 也行）
2. 用另一台电脑打开 VLC 播放器，输入 RTSP 地址拉流
3. 用手机同时拍下"电脑屏幕"和"秒表"的照片
4. 在照片上对比：
   - 秒表显示的时间是**真实时间**
   - 电脑屏幕上的画面里也有一个秒表，但那是**延迟后的时间**
   - 两个时间的差值 = 推流延迟

例如：秒表显示 15.500 秒，电脑画面里的秒表显示 15.350 秒，差值 = 150ms。

**为什么用秒表？** 因为秒表一直在走，而且数字清晰可见，很容易对比。

**精度**：大约 ±50-100ms（取决于拍照时机和肉眼判断）

#### 方法二：NTP 时间同步法

**原理**：
1. 板子和电脑都连接到同一个时间服务器（NTP 服务器），同步系统时间
2. 板子端在发送每一帧之前，把当前时间戳写入帧的元数据
3. 电脑端收到每一帧时，记录当前的系统时间
4. 延迟 = 电脑端时间 - 板子端时间戳

**优点**：精度高（可以到 10ms 以内）
**缺点**：需要修改程序代码，实现比较复杂

#### 方法三：VLC 感知验证法

**详细操作**：
1. 人在摄像头前做一个明确的动作（比如拍手）
2. 同时观察 VLC 播放器里的画面
3. 凭感觉判断"拍手"这个动作在屏幕上延迟了多久

**优点**：最简单，最直观

**缺点**：非常不精确，误差可能达到 500ms 以上

### 我们计划采用的方法

我们选择**硬件同步法（方法一）**。为什么？

1. **不需要修改代码**：直接使用现有的 RTSP 服务
2. **精度够用**：±100ms 对于验证 200ms 目标是足够的
3. **操作简单**：只需要一个秒表和一个 VLC 播放器

#### 具体步骤：

```
准备阶段：
1. 确保板子 RTSP 服务已启动（enable_rtsp: true）
2. 板子和电脑在同一个局域网

测试阶段：
1. 在手机/平板打开一个计时器 APP，放在摄像头前面
2. 电脑打开 VLC → 媒体 → 打开网络串流
3. 输入：rtsp://192.168.1.200:8554/edge_camera
4. 点击播放
5. 等待画面稳定后，用手机拍一张照片（同时拍到电脑屏幕和秒表）
6. 在照片上计算：秒表时间 - 画面时间 = 延迟
7. 重复 5-10 次取平均值
```

### 测试结果

当前 RTSP 端口（8554）未监听，RTSP 服务没有成功启动。

**原因**：RTSP 服务依赖视频编码器启动成功，但编码器初始化需要 Pipeline 的输出线程正常工作，而输出线程目前被队列阻塞影响。

**这是一个"鸡生蛋、蛋生鸡"的问题**：Pipeline 需要先稳定运行，RTSP 才能启动；但 RTSP 是 Pipeline 的输出环节之一，它也会影响 Pipeline。

### 结论

| 指标 | 状态 |
|------|------|
| RTSP 服务启动 | ❌ 端口未监听 |
| 推流延迟 | ⏳ 待调试 Pipeline 后重新测试 |

## 实验 5：系统启动时间测量

### 这实验要测什么？

验证说法："**系统启动时间从 15s 降至 5s**"

**什么是启动时间？** 从板子通电开机，到我们的 edge-ai-camera 程序开始正常运行，这整个过程的时间。

### 市面上常见的测试方法

#### 方法一：systemd-analyze 分析

Linux 系统启动时，systemd（系统和服务管理器）会记录每个步骤的耗时。`systemd-analyze` 是一个内置工具，可以帮我们分析这些数据。

**三个子命令**：

1. **`systemd-analyze time`**：显示总启动时间，分为三个阶段：
   - Kernel 阶段：从开机到内核初始化完成
   - Initramfs 阶段：加载初始内存文件系统
   - Userspace 阶段：启动所有用户空间服务

2. **`systemd-analyze blame`**：列出每个服务花了多少时间，从慢到快排序。这就像"启动速度排行榜"。

3. **`systemd-analyze critical-chain`**：显示启动的**关键路径**，即"哪个服务在等哪个服务"。A 等着 B 完成，B 等着 C 完成... 关键路径上的服务决定了总启动时间。

**举例**：
```
Startup finished in 3.7s (kernel) + 1.5s (initrd) + 15.6s (userspace) = 21.0s
```
解读：内核自己花了 3.7 秒，加载初始文件系统花了 1.5 秒，启动所有用户服务花了 15.6 秒。

```
13.479s rknn-npu.service     ← NPU 服务花了 13.5 秒！
 8.363s toybrick.service      ← 板子初始化花了 8.4 秒
```
解读：rknn-npu.service 是启动最慢的服务，占了 13.5 秒。如果能优化这个服务，总启动时间就能大幅缩短。

#### 方法二：串口日志捕获

板子通常有一个串口（UART 接口），启动时会通过串口输出大量调试信息。用 USB 转串口线连接到电脑，再用 `minicom` 或 `picocom` 等工具查看输出。

串口输出的好处是可以从**第一时间**（U-Boot 启动引导程序）开始计时，比 systemd-analyze 覆盖的范围更全。

**但**需要额外的硬件（USB 转串口线），操作更复杂。

#### 方法三：秒表计时

最原始的方法：按下电源的同时启动秒表，看着屏幕上的指示灯或登录界面，停表。

**精度极差**，只能作为一种粗略验证。

### 我们实际采用的方法

我们使用 **systemd-analyze**，因为：
1. 不需要额外硬件
2. 通过 SSH 连接即可操作
3. 输出结构化、清晰易懂
4. 可以精确定位慢的服务

#### 具体步骤

```bash
# SSH 连接到板子
ssh toybrick@192.168.1.200

# 查看总启动时间
systemd-analyze time

# 查看最慢的服务
systemd-analyze blame | head -20

# 查看关键依赖链
systemd-analyze critical-chain
```

### 测试结果

#### 优化前

```
总启动时间: 21.041s
  - Kernel:     3.769s
  - Initramfs:  1.594s
  - Userspace: 15.677s

最慢的服务:
13.479s  rknn-npu.service      ← 最大瓶颈！
 8.363s  toybrick.service       ← 第二大瓶颈
 2.271s  dev-mmcblk1p4.device   ← 设备初始化，没法优化
 0.640s  edge-ai-camera.service  ← 我们的应用，启动很快
 0.302s  NetworkManager.service
```

**分析**：
- `rknn-npu.service` 占了 13.5 秒！这个服务在做什么？
  - `npu_powerctrl`：NPU 芯片上电（约 1 秒）
  - `npu_upgrade`：**NPU 固件升级检查**（约 10 秒！）
  - 这些是每次启动都要执行的

- `toybrick.service` 占了 8.4 秒
  - 运行 `toybrick-server.sh` 初始化脚本
  - 包括 GPIO 初始化、音频配置、iptables 设置等

#### 优化措施

我们做了什么优化？

**方案**：直接启动 `npu_transfer_proxy`（NPU 传输代理），跳过固件升级检查。

为什么可以跳过？NPU 固件已经存在于 `/usr/lib/firmware/npu_fw/` 目录中，从之前的启动日志可以看到固件版本是匹配的，不需要每次都重新升级。而且我们用独立程序测试过 NPU 推理，证明 NPU 功能完全正常。

**具体操作**：

```bash
# 修改 /etc/systemd/system/rknn-npu.service
# 从：
ExecStartPre=/usr/bin/npu_powerctrl -i
ExecStartPre=/bin/sleep 0.1
ExecStartPre=/usr/bin/npu_powerctrl -o
ExecStartPre=/bin/sleep 1
ExecStartPre=/usr/bin/npu_upgrade TB-RK3399ProD    ← 这行删掉！每次都要 10 秒
ExecStartPre=/bin/sleep 1
ExecStart=/usr/bin/npu_transfer_proxy

# 改为：
ExecStart=/usr/bin/npu_transfer_proxy              ← 直接启动代理
```

#### 优化后

```
总启动时间: 14.443s
  - Kernel:     3.585s
  - Initramfs:  1.656s
  - Userspace:   9.202s      ← 从 15.7s 降到 9.2s！

最慢的服务:
 7.276s  toybrick.service     ← 现在是主要瓶颈
 2.250s  dev-mmcblk1p3.device
 0.690s  edge-ai-camera.service
```

### 结论

| 指标 | 优化前 | 优化后 | 改善 |
|------|--------|--------|------|
| 总启动时间 | 21.04s | 14.44s | **-32%** |
| rknn-npu.service | 13.48s | ~1s | **-12.5s** |
| Userspace | 15.68s | 9.20s | **-6.5s** |

| 目标值 | 实测值 | 结果 |
|--------|--------|------|
| < 5s | 14.44s | ❌ 未达标 |

**为什么还没到 5 秒？**
- `toybrick.service` 还占 7.3 秒，这是下一个可以优化的目标
- `dev-mmcblk` 设备初始化约 2 秒，这是存储卡识别，难以优化
- Kernel + Initrd 约 5 秒，取决于内核配置

## 实验 7：中断响应延迟 (cyclictest)

### 这实验要测什么？

验证说法："**内核开启 PREEMPT 抢占，中断响应延迟降低约 30%**"

在解释这个实验之前，先解释几个关键概念：

**什么是中断？**
计算机在正常运行程序时，硬件（网卡、硬盘、NPU、时钟等）可能需要 CPU 处理紧急事情。硬件会发出一个"中断信号"告诉 CPU："我有急事，暂停你当前的工作，先处理我的"。CPU 收到中断后，暂停当前任务，转而处理硬件的请求。处理完后再回去继续之前的工作。

**什么是中断响应延迟？**
从硬件发出中断信号，到 CPU 真正开始处理这个中断，这之间有一个时间差——这就是"中断响应延迟"。这个延迟越小越好，延迟大会导致系统反应迟钝。

**什么是 PREEMPT（抢占）？**
Linux 内核有一个配置叫 `PREEMPT`（全称：Preemption Model）。它控制内核在处理"系统调用"（程序请求操作系统帮忙做事）时能否被中断打断。

- **CONFIG_PREEMPT_NONE**（无抢占）：内核在处理系统调用时，必须处理完才能响应中断。响应延迟最长。
- **CONFIG_PREEMPT_VOLUNTARY**（自愿抢占）：内核在"方便的时候"可以被打断。响应延迟中等。
- **CONFIG_PREEMPT**（强制抢占）：内核可以在任何时候被打断去响应中断。响应延迟最短。
- **CONFIG_PREEMPT_RT**（实时抢占）：最强的抢占模式，几乎所有内核操作都可以被打断。用于工业控制等硬实时场景。

**什么是 cyclictest？**
cyclictest 是 Linux 实时性测试的标准工具。它的工作原理是：

1. 设定一个定时器，每隔固定时间（比如 200 微秒）触发一次
2. 每次触发时记录当前时间（称为"预期时间"）
3. 等 CPU 真正开始执行时再记录一次时间（称为"实际时间"）
4. 实际时间 - 预期时间 = 调度延迟

这个延迟反映了操作系统从"该做某件事了"到"真正开始做"之间的时间差，是衡量实时性的核心指标。

### 市面上常见的测试方法

#### 方法一：cyclictest

**工作原理**（更详细的解释）：

```c
// cyclictest 的核心逻辑简化版
while (运行中) {
    // 1. 设定下一次触发时间 = 现在 + 200微秒
    clock_nanosleep(CLOCK_MONOTONIC, TIMER_ABSTIME, &next_period, NULL);
    
    // 2. 记录实际被唤醒的时间
    clock_gettime(CLOCK_MONOTONIC, &now);
    
    // 3. 计算延迟 = 实际时间 - 预期时间
    latency = (now - next_period);
    
    // 4. 记录到直方图
    histogram[latency]++;
    
    // 5. 设定下一次触发时间
    next_period += 200us;
}
```

**参数解释**：
- `-l10000`：跑 10000 次循环。为什么是 10000？太少样本量不够，太多太慢。10000 次 × 200us = 2 秒
- `-m`：锁定内存页面（mlockall）。普通程序的内存可以被交换到磁盘（swap），但实时任务不行——因为从磁盘读回来太慢了。`-m` 参数锁定内存，防止被换出
- `-Sp90`：设置调度策略为 SCHED_FIFO，优先级 90（最高 99）。SCHED_FIFO 是先入先出实时调度
- `-i200`：间隔 200 微秒。200us 对应 5000Hz 的中断频率，属于高频场景
- `-h400`：直方图范围 0-400 微秒。超过 400us 的延迟会被记录为"溢出"
- `-q`：安静模式，不打印中间的详细信息，只输出最终结果

#### 方法二：Xenomai 实时扩展

Xenomai 是一个"实时内核扩展"，它在 Linux 内核之上添加了一层实时调度，可以实现比原生 Linux 更低的延迟（通常 < 10us）。它常用于：
- 工业机器人控制
- CNC 数控机床
- 高精度数据采集

但需要安装专门的补丁和驱动，配置复杂。

#### 方法三：ACPI PET 分析

内核自带了一个"性能事件计时器"（Performance Event Timer），可以记录每次调度事件的精确时间。但使用门槛高，需要了解内核调试接口。

### 我们实际采用的方法

我们使用 cyclictest，因为它是 Linux 实时性测试的行业标准。

#### 具体步骤

```bash
# 1. 以 root 权限运行（cyclictest 需要实时调度权限）
sudo cyclictest -l10000 -m -Sp90 -i200 -h400 -q
```

```
│ CPU0  │ 10 us   │ 28 us   │ 163 us  │ A53 小核，负载正常   │
│ CPU1  │ 12 us   │ 26 us   │ 3195 us │ A53 小核，有异常峰值 │
│ CPU2  │ 12 us   │ 26 us   │ 151 us  │ A53 小核，负载正常   │
│ CPU3  │ 11 us   │ 25 us   │ 106 us  │ A53 小核，负载正常   │
│ CPU4  │ 7 us    │ 19 us   │ 95 us   │ A72 大核，最好       │
│ CPU5  │ 6 us    │ 18 us   │ 37 us   │ A72 大核，最好       │
└───────┴─────────┴─────────┴─────────┴──────────────────────┘
```

**如何理解这些数据？**

- **Min（最小延迟）**：最好的情况。CPU4 只有 7 微秒，意味着调度非常及时
- **Avg（平均延迟）**：典型情况。A72 大核（CPU4/5）平均 18-19us，比 A53 小核（25-28us）快约 30%
- **Max（最大延迟）**：最坏情况。CPU1 出现了一次 3195us 的异常峰值，这可能是某个特定中断或设备驱动造成的

**为什么 A72 大核延迟更低？**
- A72 是高性能核心，主频 1.8GHz
- 当前配置中，A72 大核上的负载较少（推理任务还没启动）
- A53 小核上有更多的系统任务（网络、日志、中断处理等）

**CPU1 的异常峰值是怎么回事？**
3195 微秒 = 约 3.2 毫秒。这在实时系统中是比较大的延迟。可能原因：
- 某个设备驱动程序在处理中断时关闭了中断（interrupt disable）
- 系统服务在 CPU1 上做了较长时间的操作
- 这是一个偶发事件（10000 次中只有 2 次溢出）

### 结论

| CPU | 平均延迟 | 分析 |
|-----|----------|------|
| CPU0-3 (A53) | 25-28 us | 负载较高，延迟较大 |
| CPU4-5 (A72) | 18-19 us | 延迟最低，适合实时任务 |

**重要发现**：
- A72 大核平均延迟 18-19us，比 A53 小核快约 30%
- 这验证了"推理任务应该跑在 A72 大核上"的设计决策
- CPU1 有偶发异常峰值，需要关注

**待验证**：
- 需要对比 CONFIG_PREEMPT_VOLUNTARY 配置的 cyclictest 结果才能验证"30% 降低"的说法
- 当前内核可能已经是 PREEMPT 配置（因为延迟已经很低）
